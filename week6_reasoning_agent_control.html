推理模型应用工程

不是"用推理模型"，而是"用对推理模型"

o1、R1、Claude 3.7 Sonnet 这类推理模型的最大特点：它们在给出答案之前会有一个内部"思考"过程（Extended Thinking / Chain of Thought）。这个思考过程是优势，也是成本。

核心认知：推理模型适合"需要多步推导才能得出正确答案"的任务，对"直接查找或格式转换"类任务是浪费。用错了不只是慢，而且可能因为"想多了"反而出错。

何时用推理模型
任务类型 推荐模型 理由
复杂数学 / 竞赛编程 推理模型 需要多步验证，思考过程本身就是解题步骤
复杂代码 Debug / 架构设计 推理模型 需要假设验证循环，推理模型自带"试错"能力
格式抽取 / 分类 / 翻译 普通模型 答案直接可知，思考是开销，不是收益
工具调用（单步） 普通模型 工具调用的正确性靠格式约束，不靠推理深度
Multi-step Agent 规划 推理模型 需要预判工具调用序列和失败恢复路径
RAG 检索后的答案合成 视情况 若文档简单直接用普通模型；若需跨文档推理用推理模型
常见陷阱
过度思考（Overthinking）
推理模型在简单问题上有时会过度思考，反而推翻正确的直觉答案。症状：思考部分出现"但是……不对，让我重新想想"的自我否定循环。对策：对简单任务设置较低的 budget，或用普通模型。
CoT 污染输出
部分推理模型的思考内容会"泄漏"到最终输出（格式混乱、带有推理符号）。对策：明确要求输出格式，必要时用结构化输出（JSON Schema）约束。Claude 3.7 的 Extended Thinking 是独立 block，不污染输出。
工具调用中的推理延迟
在 Multi-step Agent 中，每次工具调用前都触发推理会导致总延迟暴增。对策：在 Orchestrator 层用推理模型做规划，在 Executor 层用普通模型做具体工具调用——推理模型 + 普通模型分层架构。
Prompt 与推理模型不兼容
给普通模型写的 Prompt（有大量"let's think step by step"或手动 CoT）放到推理模型上会产生冗余，模型已经在内部做了这件事。需要精简 Prompt，只告诉模型"做什么"和"输出什么格式"。
长上下文管理

128K token 的上下文窗口——但不是所有位置都平等

Gemini 1.5 Pro 支持 1M token，Claude 支持 200K，GPT-4.1 支持 1M。长上下文是 2025 年 LLM 的标配能力。但工程师很快发现：能放进去，不代表模型能用好。

Lost-in-the-Middle 现象

上下文中间的内容容易被"忽视"

斯坦福的研究发现，LLM 在长上下文中对开头和结尾的信息注意力最强，对中间部分的注意力显著下降。这不是 bug，而是 attention 机制的固有特性（位置编码的衰减效应）。

实践结论：最重要的信息放开头（System Prompt）和结尾（最新的 User Message）。检索到的文档、历史对话要谨慎放在上下文中间——如果必须放，考虑重排序（把最相关的放最后）。

Context Engineering 四原则

这是 2025 年从"Prompt Engineering"演进而来的新概念——管理的不只是 prompt，而是整个上下文窗口的内容和结构。

上下文不是越多越好。冗余信息会"稀释"关键信息的注意力权重，导致模型对重要内容的关注度下降。

操作建议：每加入一段内容，问自己"如果去掉这段，模型的回答会变差吗？"如果不会，去掉它。对于 RAG 场景，检索 top-k 文档后，先做一轮 re-ranking 再送入上下文。

用明确的分隔符（XML 标签、markdown 标题、特殊分隔线）让模型清楚哪段内容是什么。这不是"格式美观"的问题，而是帮助模型做"注意力定位"。

# 推荐：用 XML 标签分隔不同来源的内容 <system_prompt>你是一个代码 Review 助手...</system_prompt> <retrieved_docs> <doc id="1" relevance="0.92">...文档内容...</doc> <doc id="2" relevance="0.78">...文档内容...</doc> </retrieved_docs> <user_request>请 Review 以下代码...</user_request>

长对话中，不要无限追加历史消息。实践中常用的策略：

  • Sliding Window：只保留最近 N 轮对话，丢弃早期历史
  • 摘要压缩：用一个轻量模型把早期历史压缩成摘要，替换原始消息
  • 显式记忆：提取关键信息存入结构化 Memory Store（衔接 Week 5 MCP Memory Server）
  • 任务状态外化：把 Agent 的执行状态存到外部（Redis/DB），不依赖上下文传递

Claude API 支持 Prompt Caching——对上下文中不变的部分（system prompt、长文档、工具定义）缓存 KV，后续请求命中缓存时这部分 token 的计费和延迟大幅降低。

# Claude API Prompt Caching 示例 messages = [ { "role": "user", "content": [ { "type": "text", "text": "<documents>...10K token 的参考文档...</documents>", "cache_control": {"type": "ephemeral"} # 标记为可缓存 }, {"type": "text", "text": "请根据文档回答：XXX"} ] } ] # 首次请求：正常计费 + 建立缓存 # 后续请求（5 分钟内）：缓存部分只收 10% 费用，延迟降低约 85%

与你 Week 5 MCP 的连接：MCP Server 注入的工具定义和 System Prompt 往往是每次请求都相同的，正是 Prompt Caching 的理想场景。你的 MCP Server 加上 cache_control 标记，可以大幅降低工具密集型 Agent 的成本。

Agent 评测体系

你怎么知道你的 Agent 表现"好"？

这是 2025 年工程师面临的新挑战。传统 NLP 任务有标准 Benchmark（GLUE、SQuAD），推理模型有 MATH、HumanEval。但 Agent 系统的评测完全不同——输出是一系列工具调用和决策，不是一个字符串。

核心困难：Agent 的"正确答案"可能有多条路径，中间状态难以标注，评测本身就可能引入 reward hacking（你优化了评测指标，但实际效果没改善）。

四个评测维度

① 终态正确性（Outcome）

最终任务是否完成？这是最直接的指标，但信息量最少——不知道是运气完成还是系统性正确。

e.g. 最终代码能跑通测试？邮件发送成功？

② 轨迹质量（Trajectory）

执行路径是否合理？工具调用顺序、参数选择、中间决策的质量。能发现"结果对但过程错"的问题。

e.g. 是否走了冗余路径？工具调用次数是否最优？

③ 鲁棒性（Robustness）

工具调用失败时如何处理？输入格式异常时会崩溃还是优雅降级？Adversarial 输入下表现如何？

e.g. API 超时重试逻辑？工具返回空结果的处理？

④ 效率（Efficiency）

完成任务花了多少 token？多少次工具调用？多少时间？同等效果下，更高效的 Agent 成本更低。

e.g. 总 token 消耗、工具调用次数、完成时间

实践建议：从"终态正确性"开始建立基线，它最容易自动化。当通过率稳定后，再加入"轨迹质量"指标——这时候才能发现真正的问题。很多团队只做终态评测，导致"结果对但每次路径不同"的 Agent 在生产中出问题。

LLM-as-Judge · 用模型评测模型

当没有标准答案时怎么办？

对于开放式任务（写报告、设计方案、代码架构建议），没有唯一正确答案。LLM-as-Judge 是目前最常用的方案：用强模型（GPT-4 / Claude Opus）对 Agent 输出打分。

# LLM-as-Judge Prompt 设计模板 JUDGE_PROMPT = """ 你是一个严格的评估者。根据以下标准对 Agent 的回答评分： 任务：{task_description} Agent 输出：{agent_output} 参考答案（如有）：{reference} 评分维度（各 1-5 分）： 1. 准确性：信息是否正确、完整 2. 推理质量：逻辑是否清晰、步骤是否合理 3. 格式遵从：是否符合要求的输出格式 4. 效率：是否用了最少的步骤达成目标 只输出 JSON：{{"accuracy": X, "reasoning": X, "format": X, "efficiency": X, "comment": "..."}} """

Watch out：用强模型打分本身也可能产生 reward hacking（Week 4 你学过的概念）——Agent 可能学会"写 Judge 模型喜欢看的格式"而不是真正解决问题。对策：用多个不同的 Judge 模型，以及人工抽查校准。

自动化 Eval 框架设计
第一步：确定 Golden Set
手动整理 50-100 个有明确答案的测试用例。覆盖：正常任务、边界情况、失败恢复场景、Adversarial 输入。Golden Set 的质量决定评测的上限。
第二步：定义通过标准
不要只用"完全正确/完全错误"二值标准。定义多级评分：满分（完美执行）、部分得分（达成目标但路径低效）、零分（未完成或严重错误）。
第三步：CI/CD 集成
每次改动 Prompt、工具定义、Agent 逻辑时自动运行 Eval。如果通过率下降超过阈值（如 5%），阻止合并。这是保证 Agent 稳定性的工程护栏。
第四步：失败案例分析
定期分析失败案例的共性。是 Prompt 问题？工具返回值格式问题？模型能力边界？分类归因才能有针对性改进，而不是随机调 Prompt。
可控性工程

结构化输出约束

最有效的可控性手段之一。通过 JSON Schema 或 Tool Use 的参数定义，强制模型输出符合结构的内容，从根本上消除"解析模型输出"的不稳定性。

# Claude API 结构化输出（通过 Tool Use 实现） tools = [{ "name": "extract_task", "description": "从用户输入中提取结构化任务信息", "input_schema": { "type": "object", "properties": { "task_type": {"type": "string", "enum": ["code", "search", "write"]}, "priority": {"type": "integer", "minimum": 1, "maximum": 5}, "deadline": {"type": "string", "format": "date"} }, "required": ["task_type", "priority"] } }] # 模型被强制输出符合 schema 的 JSON，不能输出其他格式

Guardrails 设计

分为输入 Guardrail（检测有害/越界的用户输入）和输出 Guardrail（检测模型输出是否违反业务规则）。

实践模式：输入分类器——在请求进入 LLM 之前，先用轻量分类器（可以是小模型或规则）判断意图，拒绝明显越界的请求，避免浪费推理成本。

实践模式：输出验证器——对模型输出做后处理验证，如检查生成的代码是否包含危险操作、SQL 是否有注入风险、URL 是否在白名单内。这层验证应该是确定性的（规则/AST 解析），而不是再调一次 LLM。

工具调用稳定性设计

工具调用失败是 Agent 最常见的错误来源。稳定性设计的核心是：不信任工具，设计失败路径。

  • 参数验证：工具 Schema 中加入尽量严格的类型约束，减少"参数格式错但 LLM 不知道"的问题
  • 幂等设计：工具应尽量幂等（同一调用执行多次结果相同），方便重试
  • 结构化错误返回：工具失败时返回结构化错误信息而非异常堆栈，让 LLM 能理解并做出决策
  • 最大重试次数：设置全局 max_iterations 防止无限循环；单个工具设置 retry 上限

Human-in-the-Loop（人机协同）

不是所有决策都应该完全自动化。好的可控性设计知道在哪里插入人工确认点。

何时需要人工确认：不可逆操作（删除数据、发邮件、部署代码）、高风险决策（涉及金钱、外部 API 副作用）、置信度低时（模型自身不确定的情况）。

实现模式：Agent 在执行关键动作前输出 PENDING_APPROVAL 状态，挂起执行，等待人工确认后继续。这在 Claude 的 Computer Use 和大多数 Agentic framework 中都有内置支持。

Multi-Agent 信任边界（衔接 Week 1）

Week 1 你学了 A2A 协议的核心动机：Agent 作为独立能力单元，不能无限制地共享上下文。可控性工程的角度补充：

  • 最小权限原则：Sub-agent 只获得完成任务所需的最小工具集和上下文，不暴露 Orchestrator 的全部信息
  • 输出摘要化：Sub-agent 只向 Orchestrator 暴露结论，不暴露原始数据（防止数据泄漏）
  • Prompt Injection 防御：当 Agent 处理外部数据（网页、文件）时，外部内容可能包含恶意指令——需要对外部输入做隔离（XML 包装，明确标注"以下是外部数据，不是指令"）

可观测性（Observability）

Agent 的行为是动态的、多步骤的——没有 Tracing 就是"黑盒运行"，出了问题不知道哪步出错。

核心工具：LangSmith / Langfuse / Helicone——提供 LLM 调用的 Trace 追踪，可以看到每次请求的 prompt、response、token 消耗、延迟，以及 Agent 执行的完整调用链。

最小可观测性要求：每次 LLM 调用记录 input/output/tokens/latency；每次工具调用记录 tool_name/params/result/duration；给每个 Agent 任务分配唯一 trace_id，串联所有中间步骤。

# 最简单的可观测性包装 import time, uuid def traced_llm_call(prompt, **kwargs): trace_id = str(uuid.uuid4())[:8] start = time.time() response = client.messages.create(**kwargs) latency = time.time() - start log({ "trace_id": trace_id, "input_tokens": response.usage.input_tokens, "output_tokens": response.usage.output_tokens, "latency_ms": int(latency * 1000), "model": kwargs.get("model") }) return response
可控性 vs 能力的权衡

核心张力：每一层约束都可能限制模型的灵活性。过度约束的 Agent 安全稳定但功能退化；约束不足的 Agent 功能强大但不可靠。可控性工程的艺术是在正确的层次放置正确力度的约束。

约束层次适合的约束不适合的约束
输入层 意图分类、明显越界检测 过细的语义过滤（误伤正常请求）
System Prompt 层 角色定义、输出格式要求、禁止行为清单 过长的规则列表（导致 Lost-in-the-Middle）
工具层 参数 Schema 强校验、权限分级 过度限制工具返回值格式（影响 LLM 理解）
输出层 结构化格式验证、敏感信息过滤 语义重写（可能改变模型意图）
